ADV-CV · FPGA · REAL-TIME ▾
October 2026 YOLO upgrade + GPU deep + supervisor meeting

📖 Study — what to cover

  • YOLO architecture — single-shot detection: divides image into S×S grid, each cell predicts B bounding boxes + class probabilities simultaneously. No separate region proposal step — that's why it's fast. Focus on YOLOv8 which has the cleanest PyTorch API.
  • Adapting YOLO for security — your binary images aren't "natural" images with discrete objects. Reframe the task as classification over image regions: can YOLO detect structural zones (header region, code section, data section) in the binary visualization that correlate with malware family?
  • Semantic segmentation — U-Net architecture: encoder (downsampling) + decoder (upsampling with skip connections). Can you segment anomalous byte regions in a binary visualization? Useful as future work even if you don't fully implement it.
  • Temporal frame analysis — frame differencing: subtract consecutive network state snapshots to detect behavioral changes. Optical flow concept for tracking movement across frames. Relevant for simulating real-time streaming input.
  • Tensor Cores — NVIDIA hardware units that accelerate matrix multiplication in FP16/BF16. torch.backends.cudnn.benchmark = True enables them automatically. How torch.compile() in PyTorch 2.x further accelerates this.

November 2026 Pynq-Z2 setup + FPGA first inference + literature review

📖 Study — what to cover

  • Vitis HLS basics — writing a C++ function with HLS pragmas: #pragma HLS PIPELINE II=1 (pipeline every cycle), #pragma HLS UNROLL factor=4 (unroll loops). Understanding the synthesis report: initiation interval (II), latency in clock cycles, resource utilization (LUT, DSP, BRAM).
  • Pynq-Z2 architecture — Xilinx Zynq-7020: ARM Cortex-A9 CPU (PS) + FPGA fabric (PL) on the same chip. Python runs on the ARM side and controls FPGA overlays via MMIO and DMA. The DPU (Deep Processing Unit) is a prebuilt FPGA overlay for CNN inference.
  • AXI DMA transfers — data moves CPU→FPGA via AXI DMA controller. Transfer latency must be measured and included in your end-to-end benchmark — otherwise you're only measuring FPGA compute time, not the true deployment latency.
  • Autoencoder for anomaly detection — encoder compresses input to a bottleneck latent vector, decoder reconstructs. Train on Normal NSL-KDD traffic only. At test time: high reconstruction error = anomaly. Detects zero-day attacks that supervised classifiers miss because they've never seen the pattern.
  • Kafka architecture concept — producers write events to topics, consumers read from topics, topics are partitioned for parallel processing. Understand why this is the standard for real-time security event pipelines at scale, even if you don't implement it.

💡 Why FPGA latency vs ARM CPU — not vs A100

The Pynq-Z2 DPU won't beat an A100 in absolute throughput — that's not the comparison that matters. The relevant comparison is FPGA DPU vs the ARM Cortex-A9 on the same board running the same model. The DPU will be 5–10× faster than the ARM CPU at the same power budget. That's your result: hardware-level acceleration is viable at the edge without cloud dependency. Frame it exactly this way in your thesis.

Phase 5 — Edge AI + threat detection algorithms + adversarial basics

December 2026 – January 2027 · 2 months · Skills 13 + 14 + 15 (light)

EDGE AI · ENSEMBLE · ADV-ML ▾
December 2026 INT8 quantization + autoencoder + Chapter 1 draft

📖 Study — what to cover

  • Post-training quantization (PTQ) — calibrate FP32 weights to INT8 range using a calibration dataset (a few hundred samples). Typical accuracy loss: 1–3%. Model size halved, inference ~2× faster on INT8-capable hardware. torch.quantization.quantize_dynamic is the simplest entry point.
  • Quantization-aware training (QAT) — simulates INT8 quantization during FP32 training by inserting fake quantization nodes. Recovers accuracy lost in PTQ at the cost of longer training. Use when PTQ accuracy drop is unacceptable.
  • Model pruning — unstructured (zero out individual weights, hard to accelerate) vs structured (remove entire channels/filters, directly reduces FLOPS). For FPGA deployment, structured pruning is more useful because it reduces the DPU workload proportionally.
  • IoT edge constraints — typical embedded security sensor: ~256MB RAM, ARM Cortex-A53 or similar, 1–3W power budget, no active cooling, intermittent connectivity. Your Pynq-Z2 DPU solution addresses all of these.
  • Autoencoder reconstruction error — train encoder-decoder on Normal NSL-KDD traffic only. MSE reconstruction error on Normal samples clusters near zero; attack traffic produces high error. Choose a threshold (e.g. 95th percentile of Normal validation error) to classify as anomaly.

January 2027 Ensemble + adversarial + Chapter 2 draft

📖 Study — what to cover

  • Ensemble methods for security — soft voting: average the class probability outputs of CNN and Isolation Forest, take the argmax. Why combining a visual detector (CNN) with a statistical detector (IsolationForest) catches different attack types — they fail on complementary samples.
  • FGSM adversarial attack — Fast Gradient Sign Method: x_adv = x + ε * sign(∇_x L(model(x), y)). One line of PyTorch. Perturbs input in the direction that maximally increases the loss. Shows your model can be fooled — and you document that honestly rather than hiding it.
  • Adversarial training defense — mix adversarial examples (generated at training time) into the normal training batch. Standard defense. Measure F1 on clean vs adversarial inputs before and after retraining.
  • APT kill chain (Lockheed Martin) — 7 stages: Reconnaissance, Weaponization, Delivery, Exploitation, Installation, Command & Control, Actions on Objectives. Map your detection system to the stage it operates at: Delivery (malicious binary arrives) and Installation (binary executes).
  • Federated learning concept — multiple edge nodes train locally, share gradient updates not raw data. Central server aggregates. Understand for your future work section: "A federated extension would allow distributed FPGA nodes to collectively improve detection without centralizing sensitive network data."

📖 Study — what to cover

  • NVIDIA Nsight Systems — timeline profiler that shows GPU kernel execution, CUDA API calls, CPU-GPU data transfer gaps, and idle time. Run with nsys profile python pipeline.py. The timeline reveals where time actually goes — usually not the GPU compute itself but the data transfers and CPU overhead around it.
  • NVIDIA Nsight Compute — kernel-level profiler: memory bandwidth utilization, SM occupancy, warp efficiency, compute throughput. For each bottleneck kernel identified by Nsight Systems, run Nsight Compute to understand the root cause.
  • Pinned memory and async transfers — tensor.pin_memory() enables faster CPU→GPU transfer via DMA. torch.cuda.Stream allows overlapping data transfer with GPU compute — while GPU processes batch N, CPU prepares batch N+1. Near-free throughput improvement.
  • FastAPI for ML serving — @app.post('/detect') endpoint, Pydantic input schema, uvicorn ASGI server. Why a REST API matters: it makes your prototype look like a deployable system, not a notebook. One curl command = live demo at your defense.
  • Latency decomposition — measure each pipeline stage separately: data loading, preprocessing (CPU), H2D transfer, model forward pass (GPU), D2H transfer, postprocessing (CPU), JSON serialization. The bottleneck is almost never where you expect.

⚠ Hard committee questions — prepared answers

Why FPGA instead of a Raspberry Pi 5 or Jetson Nano at the edge?

Pi 5 (ARM Cortex-A76) has no hardware accelerator for CNN inference — it runs the same ARM code as our CPU baseline. Jetson Nano has a GPU but costs 3× more and draws 5–10W vs our Pynq-Z2's ~2W under DPU load. FPGA inference is also deterministic — no OS scheduling jitter — which matters for security SLAs with hard latency requirements.

Why computer vision on network data? Traditional feature-based ML works well on NSL-KDD.

Feature-based approaches require domain expert feature selection and are brittle to adversarial feature manipulation — an attacker who knows your feature set can craft traffic that evades detection. Visual representation is obfuscation-resistant because structural byte patterns at the memory level are much harder to randomize without breaking the malware's functionality. The MalImg benchmark confirms CNN accuracy is competitive with feature-based approaches while generalizing better to novel families.

Your F1 on NSL-KDD is lower than the published Random Forest baseline. How do you justify this?

The RF baseline is optimized purely for accuracy on a static dataset with hand-engineered features and no deployment constraints. Our system accepts raw input without feature engineering and runs in [X]ms on an FPGA edge device with no cloud dependency. The accuracy difference is the direct cost of the deployment constraints we're solving — constraints the RF baseline ignores entirely. Our baseline comparison table makes this tradeoff explicit.

What happens when attackers learn about your detection system and craft evasion attacks?

This is exactly what the adversarial robustness section addresses. At ε=0.05, our F1 drops by [N]%. Adversarial training reduces this drop to [M]%. We document this honestly as a current limitation. Full adversarial robustness — including on-device FPGA adversarial retraining — is explicitly noted as future work. A system that acknowledges its failure modes is more trustworthy than one that hides them.
